01

Purpose

ScreenSaver is a personal Windows 10/11 desktop app (not a hosted service) that displays your album art as a full-screen mosaic when the machine goes idle. It scans a music folder once, caches each album's cover, and from then on tiles those covers across every monitor, crossfading one at a time. While anything is playing through Windows System Media Transport Controls (SMTC) or iTunes, the playing album rises into a central focal tile and its mosaic tile pulses.

It is built as a suite of four cooperating components plus a small launcher: the screensaver itself, an AlbumCoverFinder scan-and-settings UI, an always-on tray companion (wallpaper, lock screen, Discord presence, LAN web page, locked-deck art), and a Stream Deck plugin that paints the current cover across one, four, or nine keys. It serves one user (the operator) on one machine; everything is local, with no cloud dependency.

The concept is borrowed from the old iTunes album-art screensaver on macOS. A short video demo is on YouTube.

02

Concept

Four front-ends draw from two shared engines. A cover cache on disk and a now-playing monitor are the only state; each component reads them and renders in its own way. Settings live in one registry key so a change made in the Finder UI propagates to everything without a restart.

ScreenSaver

The .scr. One owner-drawn form per monitor: a centered grid of square tiles, one swapping at a time, with the SMTC/iTunes track as a focal cover. ScreenSaver/.

AlbumCoverFinder

WinForms scanner and settings hub. Walks a folder, extracts cover art and tags, writes the cache, and owns every tab of configuration. AlbumCoverFinder/.

Tray companion

Always-on. Desktop wallpaper, lock-screen image, Discord Rich Presence, a LAN web page, and locked-deck artwork. The sole iTunes poller. ScreenSaver.Tray/.

Stream Deck plugin

Spreads the playing cover across 1, 4, or 9 keys; each key runs its own media control on press. ScreenSaver.StreamDeck/.

A fifth project, ScreenSaver.Launcher/, is a tiny stub that exists only so the Windows screensaver picker can list us. See the deploy fold below.

The album-cover-finding pipeline

Scan once, then everyone reads the cache

AlbumCoverFinder walks a music folder recursively, and for each unique artist+album it resizes the embedded artwork to 384x384 and writes a numbered PNG plus a small JSON sidecar to %USERPROFILE%\ScreenSaverCovers\. Every other component opens that same folder read-only.

WhyPer-file PNG cache instead of a serialized blob

The pre-2026 build serialized a Dictionary<string,Image> with BinaryFormatter to a single .bin in the user profile. BinaryFormatter was removed in .NET 9, so the format had to change. A folder of plain PNGs (index.txt maps line N to N.png) is human-debuggable, deletable per cover, needs no serialization library, and lets readers lazily decode only the covers they show. See AlbumCoverFinder/CoverMgr.cs.

Deduplication is cheap-first: the scanner only computes a SHA256 of the original artwork bytes when a second cover shows the same byte length, so byte-identical art (compilations, re-releases) is skipped without hashing every file. The length and hash ride in the JSON sidecar (AlbumCoverFinder/AlbumMetadata.cs).

Deep diveCache layout, lazy decode, and iTunes-only covers

Layout under %USERPROFILE%\ScreenSaverCovers\: index.txt (one artist+album key per line, line N maps to N.png), the N.png covers at 384x384, optional N.json metadata sidecars (artist, album, year, genre, cover byte length, cover hash), and blocklist.txt (hidden keys, one per line).

Read-only consumers (screensaver, tray, plugin) construct AlbumCoverMgr(lazyLoad: true): startup records each key's PNG path without decoding, and each cover is decoded on first display and cached. The Finder loads eager because it edits and re-saves. Tags come from TagLib#, so any format it supports works (MP3, FLAC, M4A, AAC, OGG, OPUS, WMA, WAV, AIFF, APE).

Albums whose audio files carry no embedded art (the art lives only in iTunes' cache) are captured opportunistically: when the Tray polls iTunes and finds a cover that is not in the cache, it calls AlbumCoverMgr.AddCover, which appends a new PNG plus sidecar, so the album joins the mosaic after it plays once.

How the components fit

Two shared engines, one settings key

The cover cache (AlbumCoverMgr) and the now-playing monitor (NowPlayingMonitor) are the shared engines. The monitor source file physically lives in ScreenSaver/ and is compile-linked into the Tray and the plugin so there is one implementation. The genre/year filter and every other preference live in HKCU\SOFTWARE\Demo_ScreenSaver, written by the Finder and re-read by the others (the Tray re-reads once a minute).

WhyOne linked monitor source, not a shared library or three copies

The SMTC and iTunes logic is delicate (see the now-playing deep dive) and must behave identically everywhere. Rather than a shared assembly (extra project, versioning) or copy-paste (drift), the csproj for the Tray and the plugin pull the file in with <Compile Include="..\ScreenSaver\NowPlayingMonitor.cs" Link="..." />. One source, three assemblies: a fix on the screensaver side reaches the others for free. The monitor takes a Func<string,Image> cover lookup rather than a hard dependency on the cache, which is what makes the link clean.

Now playing

SMTC for everything, a single iTunes poller for iTunes

For any app that publishes a media session (Spotify, Apple Music, foobar2000, VLC, browsers) the monitor subscribes to SMTC and reports the active track only while it is actually playing. iTunes for Windows does not publish to SMTC, so it is read over COM, but by exactly one process: the Tray polls, and the screensaver and plugin watch a pair of files the Tray writes.

WhyOnly the Tray touches iTunes COM, and never launches it

The Store build of iTunes does not expose its automation object to outside processes (GetActiveObject returns MK_E_UNAVAILABLE), so the only way to read it is CreateInstance, which launches iTunes if it is not already running. A naive poll resurrected iTunes after the user closed it and blocked its quit with "a script is using it." The fix (in NowPlayingMonitor.PollItunes): never attach unless iTunes.exe is already running, kill any instance a poll accidentally spawns during the shutdown race, release every COM reference and force a GC each poll. (An earlier build also stood aside entirely while an iPhone/iPad was connected, assuming the COM poll disturbed device sync; that turned out to be a physical USB fault — Device Descriptor Request Failed, fixed by reseating the cable — so the Tray now keeps polling during a sync and the deck stays live.) Funneling all of this through one process means iTunes' single-threaded apartment is hit once, not three times. Full history in .claude/notes.md.

Deep diveThe IPC files and why the PNG is written before the JSON

The Tray (the writer) publishes two files in %LOCALAPPDATA%\ScreenSaver\: nowplaying.png (the current cover) and nowplaying.json ({title, artist, album}). Readers run a FileSystemWatcher on the JSON. The PNG is written first, then the JSON, because a reader without its own cover cache (the plugin) pairs the JSON's metadata with the PNG on disk: if the JSON landed first, the deck would render one track behind. Both files are written temp-then-move so no reader sees a half-written file.

Each source records its last value; RecomputeAndRaise makes iTunes win when present and SMTC fill in otherwise. NowPlayingInfo.FromItunes tags the source so the plugin can show iTunes art exclusively while the screensaver shows whatever is playing.

Rendering

The screensaver is owner-drawn, with no child controls

Each monitor gets one ScreenSaverForm that paints the whole mosaic, the focal cover, and every caption itself, into a single back-buffer bitmap that is blitted in one pass. There are no child tile controls.

WhyOwner-draw replaced child PictureBoxes after a black-screen bug

On this machine's .NET 10 runtime, child controls of the screensaver form never composite to the screen: the form's own surface paints, but its children (any control type, any DPI mode, any window style) do not. The mosaic was therefore re-implemented as owner-draw. FadingPictureBox.cs now holds FadingTile, a plain renderer (not a Control) with the same transition math and a Paint(Graphics, Rectangle) method. A single 16 ms animation timer advances transitions and highlight pulses, invalidating only the dirty tile rects so a 4K surface is not cleared every frame. The full investigation is in .claude/notes.md (2026-05-29).

Deep diveGrid, transitions, gestures, and focus

The grid starts at the top-left. With a configured column count, tile size is screen width / columns; at the default (CoversWide 0) tiles are 256 px and the column count is ceil-rounded so the edges clip a partial tile rather than leave a black bar. Four transition effects (Blink, FadeToBlack, Merge, Flip) live in FadingTile; the swap cadence is GapBetweenTransitionsMs + TransitionDurationMs (Flip, 3000 ms, gap 1000 ms by default, so roughly one swap every four seconds).

Gestures depart from a stock screensaver: a mouse move over 5 px raises a manage overlay (cursor plus a hint and the hovered album's caption) rather than exiting; left-click a tile to exit; Shift-click to pin a tile as the focal cover; right-click to hide an album (it joins blocklist.txt and is replaced); any key exits. Because a saver launched indirectly is denied foreground, OnShown grabs keyboard focus with the AttachThreadInput trick, and WndProc tears down on any key while still passing the message to DefWindowProc so media/volume keys keep their system action on the way out.

The Stream Deck plugin

One action becomes one tile of an NxN cover wall

The user drops the same Cover Tile action on 1, 4, or 9 keys and tells each key its grid size, its (row, column), and what media control to run on press. Every instance subscribes to one shared monitor, crops its slice of the cover, and pushes a 144x144 PNG to its key. The plugin mirrors iTunes only.

WhyPages are bundled profiles, and a key-press dismisses the saver by PostMessage

The Stream Deck SDK never lets a plugin flip pages of a user profile; the only navigation is switchToProfile, restricted to profiles bundled with the plugin. So "pages" are four single-page bundled profiles, "Covers 1" through "Covers 4", and the Next/Previous-page actions switch between them by name with wrap-around.

A deck press cannot reach the screensaver through the input system (SMTC successes are API calls; injected media keys are grabbed system-wide by the SMTC service; a plain injected key only lands on the foreground window, which the saver routinely is not). So on every press the plugin enumerates the saver's visible top-level windows and PostMessages WM_KEYDOWN directly; PostMessage ignores focus and the saver exits on any key. No IPC handshake, no saver-side change. History in .claude/notes.md.

Deploy

Registry pointer plus a System32 stub, not a self-contained .scr

The real screensaver is a framework-dependent app under %LOCALAPPDATA%\ScreenSaver\App\. Windows is pointed at it through HKCU\Control Panel\Desktop\SCRNSAVE.EXE, and a tiny launcher in System32 restores the picker dropdown entry.

WhyA 55 MB self-contained .scr in System32 tripped antivirus

The earlier deploy copied a compressed, self-contained, single-file .scr into System32. That combination (large compressed payload, System32, unsigned) reads as a packer to Bitdefender and got blocked. The current model publishes the screensaver framework-dependent (a small exe plus a few DLLs, which looks like any .NET app) into %LOCALAPPDATA% and points SCRNSAVE.EXE at it: idle activation runs it with no admin. Because the Windows picker only lists .scr files in System32, ScreenSaver.Launcher drops a ~160 KB framework-dependent, uncompressed stub at System32\ScreenSaver.scr that reads HKCU\SOFTWARE\Demo_ScreenSaver\ScreenSaverPath and forwards the args. That one file needs UAC; everything else is per-user. See the install skill and each csproj for the publish flags.

Deep diveLocked-deck artwork over raw USB HID

When Windows locks, the Elgato app sleeps the deck and stops dispatching to plugins, so the websocket path cannot keep the cover on the keys. The Tray's LockScreenDeck instead talks to the deck over raw USB HID via StreamDeckSharp, which bypasses the Elgato app entirely; during the locked window it is effectively the sole writer. On SessionLock it opens the deck, reads LockDeckMode, and on a timer composes a full-deck frame (the cover centered and letterboxed over a blurred, darkened fill), slices it per key, and pushes each slice. On SessionUnlock it disposes the board so the Elgato app reclaims it. Diagnostics land in %LOCALAPPDATA%\ScreenSaver\deck-lock.log. The technique was reverse-engineered from BarRaider's screensaver plugin (notes.md, 2026-06-17).

QA & testing

How to run it

Prerequisites: Windows 10 build 19041 (May 2020) or newer (required for SMTC and the lock-screen API) and the .NET 10 SDK. Build with the build skill, install with the install skill (both under .claude/skills/). Installation is per-machine, no external installer.

powershell
# from the repo root
git clone https://github.com/BlashyrkhRavenDark/ScreenSaver.git
cd ScreenSaver
# build (see the build skill for the full five-output sequence)
dotnet build ScreenSaver.sln -c Release -nologo
# install onto this machine (asks which components; one step needs UAC)
# run the install skill, then pick ScreenSaver in Windows screensaver settings

The install skill deploys the screensaver to %LOCALAPPDATA%\ScreenSaver\App\ and points SCRNSAVE.EXE at it, installs the Tray with autostart, the plugin into the Elgato folder, an AlbumCoverFinder Start-menu shortcut, and (with UAC) the System32 picker stub.

How to test it, basic to advanced

Verification is split. A black-box xUnit project (ScreenSaver.Tests) now pins the deterministic logic and the renderer geometry; the manual, log-driven levels below cover what no headless test can reach (on-screen compositing, now-playing, the deck, iTunes hygiene). The manual levels go from cheapest to gnarliest.

AutomatedxUnit logic and geometry harness

From the repo root, dotnet test ScreenSaver.Tests runs the whole suite (67 cases, well under a second). It is a Floor-tier, black-box harness over the GUI-free logic in AlbumCoverFinder:

  • Cover filter (CoverFilter.Matches / IsEmpty): genre substring is case-insensitive; a year bound binds only when both the bound and the cover's year are > 0, so unknown-year covers always pass. The at-the-bound cases pin the < and > comparisons against off-by-one mutants.
  • Blocklist (CoverBlocklist): block, unblock, clear, duplicate rejection, and the blocklist.txt round-trip across two instances, against a throwaway temp dir.
  • Key and decade: AlbumCoverMgr.BuildKey and AlbumMetadata.Decade.
  • Settings (ScreensaverSettings): split into an in-range round-trip (Load(Save(x)) == x) and a separate clamp contract, because Save writes raw and Load clamps on read, so the round-trip is false across a clamp boundary. The round-trip uses an injectable registry path pointed at a throwaway SOFTWARE\Demo_ScreenSaver_Tests subkey (never the live key); the clamp runs through a pure Clamp helper with no registry.
  • Renderer geometry (golden master): the deterministic tile, column, and row math and the dest-rectangle grid, lifted from RenderMosaic into a pure DescribeLayout and pinned for a 4K (CoversWide 12) and a 1080p (native 256 px) canvas. The expected text came from an independent oracle and was approved once. The non-deterministic pixels stay on the manual capture below.

Two small behavior-preserving source changes open the black-box boundary: RegistryPath became an injectable parameter on ScreensaverSettings and CoverFilter (so tests never snapshot the live HKCU key), and the layout math moved into ComputeLayout / DescribeLayout (a no-op confirmed by the on-screen capture, and by the live log showing the same grid=12x7 tile=320 on a 4K screen). To add a case, drop a [Fact] into the matching *Tests.cs; goldens live in ScreenSaver.Tests\Expected\ next to their generator. ADR-001 · Rebuild on .NET 10 with TagLib# and a per-file PNG cache · accepted
Target .NET 10, read every tag through TagLib#, and drop id3lib and BinaryFormatter; each cover becomes a numbered PNG plus a JSON sidecar under ScreenSaverCovers, lazily decoded by the read-only consumers. Full record in adr/0001.
ADR-002 · Owner-draw the screensaver, no child controls · accepted
Child controls never composited on this .NET 10 box (the black screen), so the whole UI is painted in ScreenSaverForm.OnPaint into a blitted back-buffer, with FadingTile a non-control renderer and one timer invalidating only dirty rects. Full record in adr/0002.
ADR-003 · Deploy via SCRNSAVE.EXE plus a System32 stub, framework-dependent · accepted
A self-contained compressed .scr in System32 tripped antivirus, so the saver ships framework-dependent in %LOCALAPPDATA% with HKCU\Control Panel\Desktop\SCRNSAVE.EXE pointed at it and a tiny uncompressed System32 stub only to restore the picker entry. Full record in adr/0003.
ADR-004 · One iTunes COM poller; dismiss the saver by PostMessage, not IPC · accepted
Only the Tray polls iTunes (attach only when already running, reap COM each poll) and publishes nowplaying.png/.json for the others to watch; the plugin dismisses the saver by posting WM_KEYDOWN to its windows. An earlier named-event IPC handshake was built and removed to stay simple. Full record in adr/0004.
