LLM Council Verdict — ScreenSaver improvements

2026-05-27 · 5 advisors · 5 peer reviewers · 1 chairman

The question

Review what this project is about, see how it can be improved, then pick the best recommendations for improvements including new features, and implement them.

The One Thing to Do First

Make the tile square (384×384, not 384×360) and center the grid so it stops leaving dead bars on every non-multiple-of-360 monitor. Highest visible improvement per line of code — and the council unanimously agreed it was a real bug.

The Recommendation

Take the Executor's one-session list as the spine. Prioritize the universally-agreed bugs first, then layer in the lifecycle blind spots the peer review surfaced. Defer everything the Expansionist wanted to future sessions.

Implemented this session:
  • Off-by-one fix: Random.Next(count-1) → Random.Next(count)
  • Square 384×384 thumbnails (was 384×360)
  • Dynamic centered grid with target tile size — fills more of the screen, no dead bars
  • No-immediate-repeat tile picker (GetRandomEntry(HashSet<string> excludeKeys))
  • Static shared Random — kills multi-monitor seed collision and clustering
  • PictureBoxSizeMode.Zoom — preserves aspect of legacy non-square covers

Where the Council Agrees

  • Off-by-one in Random.Next(count-1) is a real, free-to-fix bug.
  • 384×360 tile is unjustified — covers are square; either bug or accident; fix it.
  • Integer-division grid math leaves dead bars on non-multiple resolutions.
  • README is broken as a front door — "Installation instructions to follow" is unhelpful.
  • 700-cover soft cap is silently truncating large libraries.

Where the Council Clashes

BinaryFormatter risk. The Contrarian (and 3 of 5 reviewers) called it a live RCE sink because of Google Drive sync. First Principles called it fake because the user has no attacker. Reality: the .bin lives at %USERPROFILE%\ScreenSaverPictures.bin — outside Google Drive — so First Principles is closer to the truth. The real reason to replace BinaryFormatter is portability (it's removed in .NET 9), and that's not blocking today.

Scope. Expansionist wanted a Microsoft Store product. Contrarian and First Principles wanted minimal surgery. Executor wanted a one-session list. Executor wins because the user explicitly said "implement them" — execution-mode beats refusal-mode and beats fantasy roadmap.

Blind Spots the Council Caught

  • GDI Image reference hygiene — assigning pb.Image = newImage without detaching the old one leaks references on long sessions. Now handled in OnFormClosed.
  • /p preview lifecycle — multiple reviewers flagged that nobody verified the preview path. New code uses the same dynamic-grid algorithm with a smaller target tile (80px) so it works inside the tiny parent HWND.
  • Cursor restoration — Cursor.Hide() is now paired with Cursor.Show() in OnFormClosed.
  • Multi-monitor Random correlation — multiple new Random() instances created in the same tick seed identically. Promoted to a single static readonly Random.

The Contrarian — full response

Refuse the scope. The two fatal flaws hiding in plain sight: (1) BinaryFormatter is a live RCE sink — the .bin lives in %USERPROFILE% and the project itself lives inside T:\GOOGLEDRIVE (sync = writable). (2) The 384×360 grid math is broken on every monitor that isn't a multiple of those numbers — 2560×1440 leaves a 256px dead bar. Plus 700 × ~540KB bitmaps ≈ 370MB of GDI handles and no test harness. Pick exactly two: kill BinaryFormatter, fix the grid math.

The First Principles Thinker — full response

The user is asking the wrong question. The real problem is "I want to look at my album art when I'm AFK." That reframe kills the BinaryFormatter RCE panic, kills the settings dialog (/c opening the scanner IS the settings dialog), and kills the multi-format / transitions / overlay scope creep. What it surfaces as actually broken: the off-by-one Random, the broken tile math on non-multiple resolutions, and the silent 700 truncation. Three fixes. Ship it.

The Expansionist — full response

This is a product waiting to be noticed. Integrate Windows SMTC so the currently-playing track's cover becomes a focal tile while the mosaic rearranges. Replace Mp3Lib/Id3Lib with TagLib# in one move — gets FLAC/M4A/OGG AND kills BinaryFormatter. Beat-reactive animation, color-palette wallpaper sync, "discover" mode. Wrap as Microsoft Store submission. Stretch to WinUI 3 → handhelds, secondary displays, Pi kiosk.

The Outsider — full response

The README is the front door, and it's locked: "Installation instructions to follow" plus a YouTube link is a dead end. I have no idea: do I download a .scr or build from source? Where do I point it at music? What does "soft cap of 700" mean for my 4,000-album library? Also the first-run UX is invisible. And 384×360 is a magic number — covers are square. As an outsider it looks like a bug.
